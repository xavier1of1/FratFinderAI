## Important Comparability Caveats

Not every metric means the same thing across the entire timeline.

- `done_jobs` is not the same as productive business output.
  - In later versions, a much larger share of jobs can complete into `review_required`, `terminal_no_signal`, or verification-only outcomes.
- the queue model changed from a large `deferred` bucket to typed lanes:
  - `blocked_provider`
  - `blocked_dependency`
  - `blocked_repairable`
- provider reliability changed by date window, so a provider conclusion from one day is not universally true.
- some reports measure crawl throughput, while others measure queue throughput or data quality.

For that reason, the most trustworthy longitudinal KPIs are:

- `complete_rows`
- `active_rows_with_any_contact`
- `active_rows_with_chapter_specific_instagram`
- `inactive_validated_rows`
- queue-lane ratios
- request outcome mix
- provider-attempt success/failure distributions

## Areas That Did Not Improve Cleanly

### 1. The bottleneck moved from provider blocking to dependency blocking

Comparing 2026-04-15 to 2026-04-27:

- `blocked_dependency_jobs` rose from `1890` to `2858`
  - delta: `+968`
  - relative change: about `+51.2%`
- current `blocked_dependency_ratio = 47.7%`

Current blocked reason mix is dominated by:

- `status_dependency_unmet`
- `dependency_wait`
- `identity_semantically_incomplete`

Verdict:

- the system is healthier than before
- but the next bottleneck is now dependency orchestration, not provider reliability

### 2. Email coverage remains weak relative to Instagram

Current live counts:

- `active_rows_with_chapter_specific_email = 142`
- `active_rows_with_chapter_specific_instagram = 1572`

Interpretation:

- Instagram enrichment has scaled much more successfully than email enrichment
- safe email discovery is still lagging far behind

### 3. Intake/source confirmation remains a significant drag

Current request totals:

- `succeeded / completed = 65`
- `failed / failed = 49`
- `draft / awaiting_confirmation = 48`
- `canceled / failed = 7`

Recent request creation by day:

- `2026-04-26`: `37` created, only `11` currently in `succeeded`, `25` currently in `draft / awaiting_confirmation`

The Virginia Tech benchmark shows the same issue qualitatively:

- many requests do not fail at fetch time
- they revert to `awaiting_confirmation`
- several crawls see records but do not promote usable chapter rows

Verdict:

- intake is still too fragile
- source confirmation and extraction/promotion logic are still suppressing successful business outcomes

## Throughput Warning: Raw Completion Volume Is Inflated

The current system has very high cumulative completion volume:

- `done_jobs = 256847`

But that number is not a clean success metric.

Current cumulative outcomes also include:

- `review_jobs = 112505`
- `updated_jobs = 2506`

Derived ratios:

- `updated_jobs / done_jobs` is roughly `1.0%`
- `review_jobs / done_jobs` is roughly `43.8%`

By comparison, on 2026-04-15:

- `updated_jobs = 1545`
- `done_jobs = 3866`
- `updated_jobs / done_jobs` was roughly `39.9%`

Interpretation:

- the system is processing far more work
- but a much smaller share of completed jobs turn into canonical updates
- completion volume alone substantially overstates business productivity

This is the single biggest reason to be careful with “throughput improved” claims.

## Overall Verdict

### Has FratFinderAI improved materially?

Yes, materially, but unevenly.

The strongest evidence of real improvement is:

- far higher chapter-level completeness
- far higher contact coverage
- much stronger Instagram capture
- much more inactive validation
- sharply lower provider-blocked queue share
- improved worker liveness

### Is the system healthy overall?

Health is mixed.

What is healthy now:

- data coverage is substantially better
- provider-blocked backlog is much lower than mid-April
- workers are active
- the national-only-contact safety guard is still holding

What is not healthy yet:

- dependency blocking is now the dominant queue bottleneck
- request intake still has too many `awaiting_confirmation` outcomes
- completion volume is heavily inflated by review-heavy processing
- provider reliability is still unstable by day and by provider
- stale workers still exist

### Bottom-line assessment

FratFinderAI is no longer in the same failure mode it was in around April 14-21.

It has improved from:

- low worker liveness
- provider-blocked paralysis
- modest data coverage

to:

- materially better coverage
- active workers
- lower provider pressure
- but a dependency-gated, review-heavy operating regime

That is real progress, but not end-state health.

## Recommendations

### Priority 1

Improve dependency unlock throughput.

Focus on:

- `status_dependency_unmet`
- `dependency_wait`
- prerequisite creation and reactivation
- reducing jobs that are safe but parked

### Priority 2

Reduce review-heavy completion inflation.

Measure and optimize:

- `updated_jobs / done_jobs`
- `review_jobs / done_jobs`
- canonical update yield per 1000 jobs processed

### Priority 3

Fix request intake/source confirmation friction.

Use the Virginia Tech audit patterns as regression targets:

- zero-record official-source crawls
- `awaiting_confirmation` reversions
- extraction/promotion failures after partial crawls

### Priority 4

Re-rank providers using downstream yield, not raw success alone.

Current evidence says:

- Bing is operationally strong right now
- SearXNG is not consistently winning on raw reliability
- DuckDuckGo should remain deprioritized
- Serper/Tavily remain non-default until quota/config is fixed

### Priority 5

Keep watching worker stability.

The system is no longer deadlocked, but stale workers are still present and liveness is not yet fully clean.

## Final Conclusion

The project has materially improved overall.

The improvement is most credible in:

- chapter coverage
- contact coverage
- inactive validation
- provider-blocked backlog reduction

The main caution is that raw field-job throughput now overstates business health. The next phase should optimize for productive yield and dependency resolution, not just more completed jobs.
