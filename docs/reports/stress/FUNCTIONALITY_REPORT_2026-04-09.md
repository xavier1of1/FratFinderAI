# Functionality Report: Stress Run Re-Audit

Date: `2026-04-09`

Run under analysis: `stress-20260409-full`

## Why This Version Replaces The Previous Report

The previous report was directionally useful, but one part of it was misleading: some rows displayed the shared provider-health probe bundle as if those were the chapter-specific queries for an individual field job. That was incorrect. The stress system uses a **global search preflight** to decide whether search providers are healthy enough to justify chapter-level fanout. Those probe queries are intentionally varied and are not the same thing as a chapter job search path.

This report corrects that by separating evidence into four classes:

- `Global preflight probe evidence` - batch-level provider-health probes used to decide whether external search should run at all.
- `Queue triage evidence` - invalid, dependency, or repair gating that prevented a job from entering wasteful work.
- `Job-level decision trace` - the actual search queries and trace events for a specific chapter job when that job truly executed chapter-specific work.
- `Outcome / impact` - what the system ultimately did with the job and how that affected downstream queue health.

## Executive Summary

The observed run window lasted **5:35:17** from `2026-04-09T03:52:30+00:00` to `2026-04-09T09:27:47+00:00`. The system entered the run with **8,766** stress-scope field jobs across **2,922** chapters and ended with **0 actionable** and **0 running** jobs. What remained at the end was not a hot failure tail: **1,879** jobs were intentionally deferred behind provider health, repair, or dependency gates, and **6,307** were blocked-invalid historical artifacts that triage removed from active execution.

The most important conclusion is that the architecture behaved correctly under stress once the queue policies were tightened. The dominant failure pressure over the six-hour window was **not** the graph runtime; it was the combination of (1) invalid historical junk already present in the queue, (2) degraded external search providers, and (3) prerequisite ordering bugs that initially let some low-value jobs stay actionable too long.

## Current Architecture Under Test

The run exercised the **implemented** backend-owned queue architecture, not the old distributed-control-plane design. The active lanes were the Request Worker, Contact Worker (`FieldJobSupervisorGraphRuntime + FieldJobGraphRuntime`), Repair Lane, and the persistence/worker tables exposed in the current architecture document. The most relevant architectural fact for this run is that queue ownership was singular and durable: the web layer was not owning these jobs, and the queue state was represented through typed relational fields and repair/defer lanes.

- The contact lane could defer work safely without losing ownership or leaving phantom running jobs.
- The repair lane could absorb semantically weak rows instead of forcing contact work to guess.
- Queue triage could cancel invalid historical rows at scale because those rows were now first-class queue records with typed states.
- The operator-visible run could be audited batch by batch through JSONL and DB snapshots.

## Deeper Analysis And Research

### 1. The Shared Probe Bundle Was Real, But It Was Global

The recurring probe bundle from `stress-20260409-post-hardstop.out` consisted of these four global health queries:
- `"sigma chi" University of Virginia instagram`
- `"delta chi" Mississippi State chapter website`
- `"lambda chi alpha" Purdue contact email`
- `"phi gamma delta" chapter directory`
Those probes were not trying to find any specific chapter record. They were deliberately varied across fraternity names, universities, and field shapes to test whether the provider stack could successfully return *anything* representative. When the provider stack failed all four, the system marked preflight as degraded and **skipped chapter-specific fanout** for affected jobs. That is why rows such as `phi-gamma-delta-main / sigma-tau-chapter-university-of-washington / find_instagram` should have been read as "zero chapter-specific external queries were attempted because the provider layer was already unhealthy," not "the system queried Sigma Chi for a Fiji chapter."

### 2. The Largest Failure Class Was Not Search; It Was Bad Historical Inventory

The final failed-reason distribution makes this clear: `ranking_or_report_row=3427`, `year_or_percentage_as_identity=2570`, `school_division_or_department=170`, `history_or_timeline_row=137`, and `award_or_honor_row=3`. That means most of the jobs that "failed" in the final database were not legitimate fraternity chapter contact attempts. They were artifacts that should never have been in the contact lane. This matters because it changes the interpretation of the stress run: a large part of the night was spent teaching the queue to **refuse** work more accurately, not chasing real chapters poorly.

### 3. Throughput Increased Even While Providers Stayed Sick

Peak throughput (`104.40 jobs/min`) did not happen because providers recovered. It happened after queue correctness patches made the system more decisive about what **not** to search. In other words, high throughput later in the run mostly reflected faster triage, safer deferral, and clearer prerequisite enforcement, not a healthier external environment. That is why some late batches show strong jobs/minute while the dominant outcomes are still deferred repair, deferred provider degradation, and terminal no-signal.

### 4. There Was No Online Reinforcement Learning Loop During The Run

The stress path used `langgraph_primary` in all 21 recorded batches and `policy_pack=default` in all 21 recorded batches. There was no evidence of live policy-pack switching, online bandit exploration, or online reinforcement learning changing search behavior mid-run. The "learning" that happened during the six hours was deterministic and human-authored: telemetry exposed failure pressure, code was patched, and the same cohort was rerun. That distinction matters because the run's improvements came from **architecture and queue policy changes**, not from a model autonomously adapting its own strategy.

### 5. What Still Consumes Risk After The Run

The final deferred queue shows the remaining risks clearly: `queued_for_entity_repair=492`, `provider_degraded=471`, `identity_semantically_incomplete=421`, `dependency_wait=281`, `transient_network=99`, `website_required=96`, `provider_low_signal=19`. The system is healthier because these are explicit, typed reasons rather than hidden thrash. But they still represent real follow-up work: a future run can only improve business yield further by either improving provider health, increasing authoritative school/chapter validation coverage, or draining the repair backlog.
